GoalShield: Executive Drift Under Interruption

### Problem Statement

Static planning benchmarks often show whether a model can produce a valid multi-step answer when all rules are presented cleanly in one place. They do not cleanly isolate executive control: preserving a goal, suppressing irrelevant but salient updates, switching to the correct rule state, and repairing a plan with minimal disruption.

GoalShield targets that gap. It asks whether a model can track a baseline schedule, identify which update packets are actually binding, and produce the smallest valid repair instead of over-replanning from scratch. The benchmark is designed to reveal a failure mode we call **executive drift**: the tendency to follow visible or recent instructions without maintaining the correct control state over the task.

### Task & Benchmark Construction

The benchmark focuses on one domain only: **schedule repair under interruption**.

Each task contains:

1. A baseline schedule over 5 to 7 abstract tasks.
2. A small set of base rules that are initially valid.
3. A stream of update packets with mixed authority.
4. An optimization objective: produce the valid final schedule that moves as few tasks as possible from the baseline schedule.

Only packets with the correct team and `status=APPLY` are binding. If two binding packets target the same rule key, the later one overrides the earlier one. This lets the benchmark probe four executive sub-functions inside a single task family:

- **Planning**: produce a schedule that satisfies all active constraints.
- **Inhibitory control**: ignore distractor packets from the wrong team or wrong status.
- **Cognitive flexibility**: switch to the current rule state when an earlier packet is overridden.
- **Working memory / goal maintenance**: retain the baseline plan and optimize for minimum-edit repair rather than forgetting the original objective.

GoalShield v1 uses three task families:

- `shield`: one real packet among distractors.
- `switch`: two real packets on the same rule key, where the later packet overrides the earlier one.
- `repair`: multiple real packets forcing a small but nonzero repair.

### Dataset

The committed v1 benchmark contains 54 tasks:

- 18 `easy` shield tasks
- 18 `medium` tasks split evenly between shield and switch
- 18 `hard` tasks split evenly between switch and repair

All tasks are synthetic and generated by code in this repository. Gold answers are fully verifiable because each task is solved exactly by enumerating all possible schedules and selecting the one that:

1. satisfies the final active rule set,
2. minimizes the number of moved positions relative to the baseline,
3. breaks ties lexicographically.

This guarantees deterministic scoring and eliminates ambiguity in the target answer.

### Technical Details

The implementation has four key components:

- A **scenario generator** that constructs baseline schedules, rule streams, distractor packets, and difficulty-controlled repair targets.
- An **exact symbolic solver** that enumerates all schedules and computes the gold answer.
- A **grading function** that scores schedule exact match, packet-selection F1, and moved-task accuracy.
- A **Kaggle Benchmarks notebook** that imports the benchmark directly from GitHub, runs a staged frontier-model sweep through the `kaggle-benchmarks` SDK, logs phase progress to `/kaggle/working/goalshield_progress.json`, and exports overall, per-difficulty, per-family, error-profile, and BenchPress novelty artifacts.

The leaderboard metric is a weighted composite:

- 55% exact final schedule match
- 20% position-wise schedule accuracy
- 15% packet-selection F1
- 10% exact moved-task count

This preserves partial signal while still rewarding exact executive control.

### Results, Insights, and Conclusions

Before running frontier models, we evaluated several deterministic heuristics as sanity checks:

| Policy | Schedule Exact | Packet F1 | Composite |
| --- | ---: | ---: | ---: |
| `latest_visible_key` | 0.438 | 0.375 | 0.504 |
| `same_team_any_status` | 0.000 | 0.646 | 0.205 |
| `apply_all_packets` | 0.000 | 0.504 | 0.175 |
| `ignore_updates` | 0.000 | 0.000 | 0.120 |

These sanity checks already show the benchmark is not reducible to a single superficial trick. The strongest naive baseline is the **recency-biased policy** (`latest_visible_key`), not the “ignore everything” baseline. That indicates GoalShield is specifically sensitive to *control-state maintenance* rather than raw constraint solving.

The central hypothesis for frontier-model runs is:

- lighter or recency-biased models will perform relatively well on `easy` shield tasks but degrade sharply on `hard` switch/repair tasks;
- stronger reasoning models will recover packet selection more reliably but may still over-edit the plan, exposing a gap between rule tracking and revision economy;
- this creates a profile not captured by standard planning or attention-only benchmarks, because success requires jointly solving rule filtering, rule switching, and minimum-edit repair.

The attached Kaggle notebook is configured to run a broad frontier-model sweep and export `goalshield_model_summary.csv`, `goalshield_difficulty_summary.csv`, `goalshield_family_summary.csv`, `goalshield_error_profile.csv`, and `goalshield_benchpress_report.json` when BenchPress can be computed.
